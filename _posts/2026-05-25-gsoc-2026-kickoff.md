Hi, I am Aman Kumar. I have been accepted into Google Summer of Code
2026 with HoloViz, under the NumFOCUS umbrella, and I still have to
say that out loud to believe it.

The project is the Lumen and xarray integration, 350 hours of it. My
mentors are Andrew Huang and Andy Maloney.

## The problem, in one sentence

A climate scientist has a NetCDF file: sea surface temperature over
latitude, longitude and time. They want to ask, in plain English,
"what was the average sea surface temperature in the Pacific in
January 2020", and get back a table and a map. With Lumen that should
be a one-liner. Today it is an afternoon of pandas glue code that
flattens the cube and throws the coordinates away on the way in.

Lumen turns a declarative spec into a data app, and
[Lumen AI](https://lumen.holoviz.org) lets you talk to your data.
Both are strongest when the data is a flat table. A lot of real
scientific data is not flat, it is N-dimensional, and the tool for
that in Python is [xarray](https://xarray.dev). Closing that gap is
the whole project.

## What we have done so far

Before GSoC officially starts on May 25, the work is already well
underway. Twenty-six of my pull requests have been merged into Lumen
in the run-up: mostly small bug fixes and quality improvements,
learning the codebase from the inside by paying down its little
papercuts. That ramp-up mattered: it is how I went from "outsider with
a proposal" to someone the maintainers were willing to merge from on a
350-hour project.

Two of those merged PRs map directly to Goals 4 and 5 of the official
proposal:

- **Goal 4. Implement an XarraySource for Lumen.** Shipped as
  **[#1741, Add XArraySQLSource for N-dimensional scientific data](https://github.com/holoviz/lumen/pull/1741)**,
  merged 26 Mar 2026. Wraps xarray Datasets from local NetCDF or
  Zarr, exposes tables and schemas to Lumen's AI agents, and uses
  xarray-sql plus DataFusion under the hood.
- **Goal 5. Extend Lumen AI agents to understand N-dimensional
  data.** Shipped as
  **[#1791, Wire XArraySQLSource into Lumen AI](https://github.com/holoviz/lumen/pull/1791)**,
  merged 17 Apr 2026. The agent side, so the climate-scientist
  question above actually returns something through chat.

So the basic loop already exists in `main`. The summer is about
making it good, and making it reach.

## What we are planning to do

The honest version: the foundation was the easy twenty percent. The
remaining work is the six other main goals from the official proposal
plus three additional goals. Ordered roughly by dependency, since the
Narwhals work blocks several of the others.

### Main goals

**Goal 1. Add a DataFrameLike param type to Param ([param#975](https://github.com/holoviz/param/issues/975)).**
A new parameter class that validates and accepts any object
conforming to the [Narwhals](https://narwhals-dev.github.io/narwhals/)
dataframe protocol (pandas, Polars, cuDF, Modin), giving the entire
HoloViz ecosystem a single, backend-agnostic way to declare "this
parameter holds tabular data."

**Goal 2. Rework Lumen pipelines to use Narwhals ([lumen#1788](https://github.com/holoviz/lumen/issues/1788)).**
Replace direct pandas and DuckDB assumptions in Lumen's Filter,
Transform, SQLTransform and Pipeline internals with Narwhals
operations so that any supported backend flows through without
conversion. This builds on the pattern already landed in HoloViews
1.22.

**Goal 3. Translate xarray-sql output back to xarray ([xarray-sql#58](https://github.com/xarray-contrib/xarray-sql/issues/58)).**
Contribute upstream to xarray-sql so query results can be round
tripped: SQL query, tabular result, reconstructed xarray Dataset with
original coordinates, dimensions and attributes. This is the inverse
of xarray-sql's current raster-to-table direction, and should
leverage the new DataFusion backend.

**Goal 6. Improve xarray and array data support in Vega-Lite
visualizations ([altair#891](https://github.com/vega/altair/issues/891)).**
Altair and Vega-Lite currently have no native xarray support, forcing
users to flatten to DataFrames and lose labeled dimensions.
Investigate and contribute to better array data handling in the
Vega-Lite data model, whether through Narwhals-based data
transformers in Altair or by generating specs that preserve
coordinate metadata, so that Lumen's VegaLite-based visualization
agent can work with xarray data without lossy conversions.

**Goal 7. JSON-serializable HoloViews plot specifications ([holoviews#791](https://github.com/holoviz/holoviews/issues/791)).**
HoloViews currently lacks a well-defined JSON serialization format,
relying on pickle or full HTML export. For the MCP use case we need a
lightweight, JSON-representable plot spec (element type, data
reference, encodings, styling) that can be transmitted over the MCP
JSON-RPC channel and reconstructed client-side, as a Vega-Lite spec,
a Bokeh JSON document, or a new HoloViews-native format. This also
benefits notebook sharing, dashboard serialization, and LLM-generated
plot editing.

**Goal 8. End-to-end integration tests with scientific datasets.**
Validate the full pipeline (load, query, transform, visualize)
against real-world datasets: a NOAA climate reanalysis Zarr on the
cloud, a microscopy image stack in local NetCDF, and an AnnData
single-cell dataset, ensuring Dask-backed lazy computation works
throughout.

### Additional goals

**Additional. STAC and Pangeo Forge catalog Source for Lumen.**
A new Lumen Source class that accepts a STAC API endpoint or a
Pangeo Forge URL and exposes each cataloged item as a queryable
Lumen table. Built on pystac-client, with intake-xarray intentionally
avoided since it is no longer actively maintained. The AI agent lists
the available datasets through the catalog, picks one by name, and
instantiates an XArraySQLSource under the hood with no further
configuration. This turns the climate-scientist onboarding flow from
"first download this 4 GB NetCDF, then write a YAML" into a single
command, `lumen-ai serve --catalog <stac-api-url>`, and gives Lumen a
path into the entire Pangeo data ecosystem without users needing to
learn intake's API.

**Additional. Coordinate-aware filtering through Tabulator and [lumen#1599](https://github.com/holoviz/lumen/pull/1599).**
Today Lumen's Filter classes target tabular columns, but for an
XArraySource the natural filterable axes are dimensions such as time,
lat, lon and pressure_level. This work picks up the draft at
lumen#1599 ("Add the ability to add filters to exploration"), which
already prototypes a UI to add and remove filters on a SQLOutput, and
extends it so the same surface also covers xarray coordinate
dimensions. Tabulator's built-in filter UI is reused as the
substrate: tabular sources keep generating SQL WHERE clauses, while
xarray sources wire through to `xarray.Dataset.sel(...)`.

**Additional. AnnData support driven through [holoviz-topics/lumen-anndata](https://github.com/holoviz-topics/lumen-anndata).**
The bioinformatics user story has a dedicated repository, lumen-anndata
("Chat with anndata"), backed by hv-anndata. Rather than duplicating
AnnData logic inside core Lumen, this work drives a few of that
repo's open issues to closure during community bonding: shipping the
`adata.chat()` entry point (#42), wiring ManifoldMapAnalysis and
Dotmap visualizations from hv-anndata into the Lumen Explorer (#6),
and surfacing generated sources in the data source controls (#27). It
also validates the full round trip against a public `.h5ad` dataset
such as scanpy's PBMC 3k tutorial, and ensures that the Narwhals and
XArraySource refactors in the main project's earlier goals do not
regress that downstream package.

I am deliberately not going to land this as one giant pull request at
the end. Each item above is its own reviewable change. Small
reviewable changes beat big impressive ones, and I would rather be
corrected early and often than be clever in private for three months.
